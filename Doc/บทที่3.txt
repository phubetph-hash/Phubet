# สรุปบทที่ 3: การออกแบบและวิเคราะห์ระบบ (แบบละเอียด)

## 📋 ภาพรวมบทที่ 3

บทนี้เป็นหัวใจสำคัญของโครงงาน เน้นการ**วิเคราะห์และออกแบบระบบ**อย่างละเอียด ครอบคลุม:
- การออกแบบกระบวนการทำงาน
- การวิเคราะห์การไหลของข้อมูล (DFD)
- การออกแบบฐานข้อมูล (ER-Diagram)
- การออกแบบหน้าจอ (UI/UX)

---

## 🎯 3.1 การออกแบบ (System Design Overview)

### 📌 แนวคิดหลัก
พัฒนาเว็บแอปพลิเคชันที่:
- **อำนวยความสะดวก** ให้นิสิตเลือกอาจารย์ที่ปรึกษา
- **ติดตามสถานะคำขอ** ได้อย่างมีประสิทธิภาพ
- **จัดการข้อมูล** ระหว่างนิสิต-อาจารย์-ผู้ดูแลระบบได้อย่างรวดเร็ว
- **เป็นระบบระเบียบ** ลดความซับซ้อนในการดำเนินงาน

### 🔄 สรุปฟังก์ชันแต่ละกลุ่มผู้ใช้

#### 👨‍🎓 นิสิต (6 ฟังก์ชันหลัก)
1. ลงทะเบียน/เข้าสู่ระบบ
2. ค้นหาและดูข้อมูลอาจารย์
3. ส่งคำขอปรึกษา (พร้อมไฟล์ PDF)
4. ตรวจสอบสถานะคำขอ
5. แก้ไขข้อมูลส่วนตัว
6. ออกจากระบบ

#### 👨‍🏫 อาจารย์ (4 ฟังก์ชันหลัก)
1. ลงทะเบียน/เข้าสู่ระบบ
2. ดูและจัดการคำขอ (อนุมัติ/ปฏิเสธ/ให้ข้อเสนอแนะ)
3. แก้ไขข้อมูลส่วนตัว (รวมจำนวนโปรเจคที่รับได้)
4. ออกจากระบบ

#### 🔧 ผู้ดูแลระบบ (7 ฟังก์ชันหลัก)
1. เข้าสู่ระบบ
2-6. จัดการข้อมูลหลัก (คณะ, ภาควิชา, สาขา, ความเชี่ยวชาญ, ปีการศึกษา)
7. แก้ไขข้อมูลส่วนตัว

---

## 🔄 3.2 การวิเคราะห์ระบบ (System Analysis)

### 📊 3.2.1 แผนภาพกระแสข้อมูล (Data Flow Diagram - DFD)

DFD เป็นเครื่องมือสำคัญในการ:
- แสดงการไหลของข้อมูลในระบบ
- วิเคราะห์กระบวนการทำงาน
- เข้าใจความสัมพันธ์ระหว่างผู้ใช้และระบบ

---

### 🌐 Level 0: Context Diagram

**แสดงภาพรวมระบบทั้งหมด**

#### ผู้ใช้และการโต้ตอบ:

**👨‍🎓 นิสิต →→→ ระบบ**
- Input: ข้อมูลลงทะเบียน, ข้อมูลเข้าสู่ระบบ, คำค้นหาอาจารย์, คำขอปรึกษา, การตรวจสอบสถานะ
- Output: ผลการลงทะเบียน, ผลการเข้าสู่ระบบ, รายชื่อ/ข้อมูลอาจารย์, สถานะคำขอ

**👨‍🏫 อาจารย์ ↔↔↔ ระบบ**
- Input: ข้อมูลลงทะเบียน, ข้อมูลเข้าสู่ระบบ, การอนุมัติ/ปฏิเสธคำขอ, ข้อเสนอแนะ
- Output: ผลการลงทะเบียน, รายการคำขอที่ได้รับ, ผลการจัดการคำขอ

**🔧 ผู้ดูแลระบบ ↔↔↔ ระบบ**
- Input: ข้อมูลเข้าสู่ระบบ, การแก้ไขข้อมูลหลัก (คณะ, ภาควิชา, สาขา, ความเชี่ยวชาญ, ปีการศึกษา)
- Output: ผลการแก้ไขข้อมูล

> 💡 **จุดสำคัญ:** Context Diagram แสดงว่าระบบมี 3 External Entities และมีการแลกเปลี่ยนข้อมูลทั้งทางเข้าและทางออก

---

### 🔍 Level 1: DFD Level 0 - กระบวนการหลัก

**ระบบแบ่งเป็น 12 Process หลัก:**

| Process | ชื่อกระบวนการ | ผู้เกี่ยวข้อง | Data Store |
|---------|---------------|--------------|------------|
| **1.0** | ลงทะเบียน | นิสิต, อาจารย์ | D1 นิสิต, D2 อาจารย์ |
| **2.0** | เข้าสู่ระบบ | ทุกกลุ่ม | D1, D2, D3 ผู้ดูแล |
| **3.0** | ค้นหาอาจารย์ที่ปรึกษา | นิสิต | D2 อาจารย์ |
| **4.0** | ส่งคำขอปรึกษา | นิสิต | D4 คำขอ, D1, D2, D9 |
| **5.0** | ตรวจสอบสถานะคำขอ | นิสิต | D4 คำขอ, D1 นิสิต |
| **6.0** | จัดการข้อมูลส่วนตัว | ทุกกลุ่ม | D1, D2, D3 |
| **7.0** | จัดการคำขอ | อาจารย์ | D4 คำขอ, D2 อาจารย์ |
| **8.0** | จัดการข้อมูลคณะ | ผู้ดูแล | D5 คณะ |
| **9.0** | จัดการข้อมูลภาควิชา | ผู้ดูแล | D6 ภาควิชา |
| **10.0** | จัดการข้อมูลสาขาวิชา | ผู้ดูแล | D7 สาขาวิชา |
| **11.0** | จัดการความเชี่ยวชาญ | ผู้ดูแล | D8 ความเชี่ยวชาญ |
| **12.0** | จัดการปีการศึกษา/ภาคเรียน | ผู้ดูแล | D9 ปีการศึกษา |

### 🔬 Level 2: DFD Level 1 - รายละเอียดแต่ละ Process

#### 📝 **Process 1: ลงทะเบียน** (แตกเป็น 3 Sub-Process)

```
1.1 กรอกข้อมูลการลงทะเบียน
    Input: ข้อมูลนิสิต/อาจารย์
    Output: ข้อมูลการลงทะเบียน

1.2 ตรวจสอบลงทะเบียน
    Input: ข้อมูลที่ต้องการตรวจสอบ
    Check: D1 นิสิต, D2 อาจารย์
    Output: ผลการตรวจสอบ

1.3 แสดงผลการลงทะเบียน
    Output: ผลการลงทะเบียน (สำเร็จ/ไม่สำเร็จ)
```

#### 🔐 **Process 2: เข้าสู่ระบบ** (3 Sub-Process)

```
2.1 กรอกข้อมูลเข้าใช้งาน
    Input: อีเมล, รหัสผ่าน

2.2 ตรวจสอบข้อมูล
    Check: D1, D2, D3
    Validate: ตรวจสอบความถูกต้อง

2.3 แสดงผลการเข้าสู่ระบบ
    Output: ผลการเข้าสู่ระบบ (สำเร็จ/ล้มเหลว)
```

#### 🔍 **Process 3: ค้นหาอาจารย์ที่ปรึกษา** (3 Sub-Process)

```
3.1 รับคำค้นหาอาจารย์
    Input: คำค้นหา (ชื่อ/ความเชี่ยวชาญ/สาขา)

3.2 ค้นหาข้อมูลอาจารย์
    Search: D2 อาจารย์
    Filter: ตามเงื่อนไขที่ระบุ

3.3 แสดงข้อมูลอาจารย์
    Output: รายชื่อและรายละเอียดอาจารย์
```

#### 📤 **Process 4: ส่งคำขอปรึกษา** (3 Sub-Process)

```
4.1 รับข้อมูลคำขอปรึกษา
    Input: 
    - ปีการศึกษา, ภาคเรียน
    - รายละเอียดโปรเจค
    - ไฟล์ PDF

4.2 ตรวจสอบคำขอ
    Validate:
    - ข้อมูลครบถ้วน
    - อาจารย์ยังรับโปรเจคได้
    - ไฟล์ถูกต้อง

4.3 บันทึกคำขอ
    Save to: D4 คำขอปรึกษา
    Status: "รอดำเนินการ"
    Output: ยืนยันการส่งคำขอ
```

#### 📊 **Process 5: ตรวจสอบสถานะคำขอ** (2 Sub-Process)

```
5.1 ตรวจสอบสถานะคำขอ
    Query: D4 คำขอปรึกษา
    Filter: ตามรหัสนิสิต

5.2 แสดงผลสถานะคำขอ
    Output:
    - สถานะ (รอดำเนินการ/อนุมัติ/ปฏิเสธ)
    - วันที่ส่ง
    - วันหมดอายุ
    - วันที่ตอบรับ
    - ข้อเสนอแนะ (ถ้ามี)
```

#### ✏️ **Process 6: จัดการข้อมูลส่วนตัว** (2 Sub-Process)

```
6.1 แก้ไขข้อมูลส่วนตัว
    Input: ข้อมูลที่ต้องการแก้ไข
    Access: D1/D2/D3 (ตามประเภทผู้ใช้)

6.2 แสดงผลข้อมูลส่วนตัว
    Update: ฐานข้อมูล
    Output: ผลการแก้ไข (สำเร็จ/ล้มเหลว)
```

#### ✅ **Process 7: จัดการคำขอ** (2 Sub-Process)

```
7.1 จัดการคำขอ
    Query: D4 คำขอปรึกษา
    Filter: ตามรหัสอาจารย์
    Display: รายการคำขอทั้งหมด

7.2 การอนุมัติหรือปฏิเสธคำขอ
    Action:
    - อนุมัติ → Status: "อนุมัติ"
    - ปฏิเสธ → Status: "ปฏิเสธ"
    - เพิ่มข้อเสนอแนะ
    Update: D4 คำขอ, D2 อาจารย์ (จำนวนโปรเจค)
```

#### 🏢 **Process 8-12: จัดการข้อมูลหลัก** (ผู้ดูแลระบบ)

แต่ละ Process มีรูปแบบเดียวกัน:
- **Input:** การแก้ไขข้อมูล (เพิ่ม/ลบ/แก้ไข)
- **Update:** ฐานข้อมูลที่เกี่ยวข้อง
- **Output:** ผลการแก้ไข

### 🔗 ความสัมพันธ์ระหว่างตาราง (Relationships)

#### 1️⃣ **Student ↔ Faculty/Department/Program**
```
- Student.faculty_id → Faculty.faculty_id (Many-to-One)
- Student.department_id → Department.dept_id (Many-to-One)
- Student.program_id → Program.program_id (Many-to-One)
```
**ความหมาย:** นิสิต 1 คนสังกัด 1 คณะ/ภาควิชา/สาขา

---

#### 2️⃣ **Advisor ↔ Faculty/Department/Program/Expertise**
```
- Advisor.faculty_id → Faculty.faculty_id (Many-to-One)
- Advisor.department_id → Department.dept_id (Many-to-One)
- Advisor.program_id → Program.program_id (Many-to-One)
- Advisor.expertise_id → Expertise.exp_id (Many-to-One)
```
**ความหมาย:** อาจารย์ 1 คนสังกัด 1 คณะ/ภาควิชา/สาขา และมี 1 ความเชี่ยวชาญ

---

#### 3️⃣ **Request ↔ Student/Advisor** ⭐ สำคัญที่สุด
```
- Request.student_id → Student.student_id (Many-to-One)
- Request.advisor_id → Advisor.advisor_id (Many-to-One)
```
**ความหมาย:** 
- นิสิต 1 คนส่งคำขอได้หลายครั้ง
- อาจารย์ 1 คนรับคำขอได้หลายคำขอ

---

#### 4️⃣ **Department ↔ Faculty**
```
- Department.faculty_id → Faculty.faculty_id (Many-to-One)
```
**ความหมาย:** ภาควิชาหลายภาคสังกัด 1 คณะ

---

#### 5️⃣ **Program ↔ Department**
```
- Program.dept_id → Department.dept_id (Many-to-One)
```
**ความหมาย:** สาขาวิชาหลายสาขาอยู่ใน 1 ภาควิชา

## 🎨 หลักการออกแบบ UI/UX

### ✨ **Design Principles**

**1. Simplicity (ความเรียบง่าย)**
- เมนูชัดเจน ไม่ซับซ้อน
- การนำทางง่าย (Navigation Bar + Breadcrumb)
- ใช้ Icon ประกอบเพื่อความเข้าใจง่าย

**2. Consistency (ความสอดคล้อง)**
- สีและรูปแบบเดียวกันทั้งระบบ
- ตำแหน่งเมนูเหมือนกันทุกหน้า
- Font และขนาดข้อความสม่ำเสมอ

**3. Feedback (การตอบสนอง)**
- แสดงข้อความยืนยันเมื่อทำสำเร็จ
- แสดง Loading Indicator ขณะรอข้อมูล
- แสดง Error Message ที่ชัดเจน

**4. Accessibility (การเข้าถึง)**
- รองรับ Screen Reader
- Keyboard Navigation
- Color Contrast ที่เหมาะสม

**5. Responsive Design**
- รองรับทุกขนาดหน้าจอ (Desktop/Tablet/Mobile)
- Touch-Friendly สำหรับมือถือ
- Hamburger Menu สำหรับหน้าจอเล็ก

## 🔐 Security Features (คุณสมบัติด้านความปลอดภัย)

### 🛡️ **มาตรการความปลอดภัย**

**1. Authentication (การยืนยันตัวตน)**
- Password Hashing (bcrypt/SHA-256)
- Session Management
- Auto Logout หลังไม่ใช้งานนาน

**2. Authorization (การควบคุมสิทธิ์)**
- Role-Based Access Control (RBAC)
  - นิสิต: ดูได้เฉพาะข้อมูลของตัวเอง
  - อาจารย์: จัดการเฉพาะคำขอของตัวเอง
  - Admin: จัดการข้อมูลหลักทั้งหมด

**3. Data Validation**
- Client-Side Validation (JavaScript)
- Server-Side Validation (PHP/Node.js)
- SQL Injection Prevention (Prepared Statements)
- XSS Protection
- CSRF Token

**4. File Upload Security**
- จำกัดประเภทไฟล์ (เฉพาะ PDF)
- จำกัดขนาดไฟล์ (5 MB)
- Virus Scan (ถ้าเป็นไปได้)
- Sanitize ชื่อไฟล์

**5. Privacy**
- ซ่อนอีเมล/เบอร์โทร บางส่วน (เช่น s***@email.com)
- Log การเข้าถึงข้อมูล
- GDPR Compliance (ถ้าจำเป็น)

## 📋 Business Logic (กฎเกณฑ์ทางธุรกิจ)

### ⚙️ **กฎการทำงานของระบบ**

**1. การส่งคำขอ**
```
✅ ส่งได้เมื่อ:
   - อาจารย์ยังรับโปรเจคได้ (project_capacity > 0)
   - ไม่มีคำขอที่รอดำเนินการกับอาจารย์คนเดียวกัน
   - ระบุรายละเอียดครบถ้วน

❌ ส่งไม่ได้เมื่อ:
   - อาจารย์รับโปรเจคเต็มแล้ว
   - มีคำขอค้างอยู่แล้ว
   - ไม่แนบไฟล์ PDF
```

**2. การอนุมัติคำขอ**
```
เมื่ออาจารย์อนุมัติ:
1. อัปเดต status → "อนุมัติ"
2. บันทึก approval_date → วันที่ปัจจุบัน
3. ลด advisor.project_capacity -= 1 ⭐
4. ส่ง Notification ให้นิสิต
5. ปิดการส่งคำขอใหม่ (ถ้านิสิตมีอาจารย์แล้ว)
```

**3. การปฏิเสธคำขอ**
```
เมื่ออาจารย์ปฏิเสธ:
1. อัปเดต status → "ปฏิเสธ"
2. บันทึก approval_date → วันที่ปัจจุบัน
3. ⚠️ ต้องระบุ suggestion (เหตุผล) - Required
4. ส่ง Notification ให้นิสิต
5. นิสิตสามารถส่งคำขอใหม่ได้
```

**4. วันหมดอายุคำขอ**
```
expire_date = request_date + 7 วัน

เมื่อหมดอายุ:
- ถ้ายัง status = "รอดำเนินการ"
  → เปลี่ยนเป็น "หมดอายุ" (Optional)
- นิสิตสามารถส่งคำขอใหม่ได้
```

**5. จำนวนโปรเจคของอาจารย์**
```
คำนวณจำนวนที่ยังรับได้:
available = project_capacity - จำนวนที่อนุมัติแล้ว

ถ้า available = 0:
- แสดงสถานะ "ไม่รับโปรเจคแล้ว" (สีแดง)
- ปิดปุ่ม "ส่งคำขอ"
- ซ่อนอาจารย์จากผลการค้นหา (Optional)
```

## 📊 Database Performance Optimization

### ⚡ **การเพิ่มประสิทธิภาพฐานข้อมูล**

#### **1. Indexing Strategy**
```sql
-- สร้าง Index สำหรับการค้นหา
CREATE INDEX idx_advisor_expertise ON Advisor(expertise_id);
CREATE INDEX idx_advisor_faculty ON Advisor(faculty_id);
CREATE INDEX idx_request_status ON Request(status);
CREATE INDEX idx_request_student ON Request(student_id);
CREATE INDEX idx_request_advisor ON Request(advisor_id);
CREATE INDEX idx_request_date ON Request(request_date);

-- Composite Index สำหรับการค้นหาแบบรวม
CREATE INDEX idx_advisor_capacity ON Advisor(faculty_id, program_id, project_capacity);
```

#### **2. Query Optimization**
```sql
-- ใช้ JOIN แทน Subquery
-- ❌ ช้า
SELECT * FROM Student 
WHERE faculty_id IN (SELECT faculty_id FROM Faculty WHERE faculty_name = 'วิทยาศาสตร์');

-- ✅ เร็ว
SELECT s.* FROM Student s
INNER JOIN Faculty f ON s.faculty_id = f.faculty_id
WHERE f.faculty_name = 'วิทยาศาสตร์';
```

#### **3. Caching Strategy**
```
📦 Cache ข้อมูลที่ไม่เปลี่ยนบ่อย:
- รายชื่อคณะ/ภาควิชา/สาขา (Cache 24 ชม.)
- รายชื่อความเชี่ยวชาญ (Cache 24 ชม.)
- ข้อมูลอาจารย์ (Cache 1 ชม.)
- Invalidate cache เมื่อมีการแก้ไข
```

#### **4. Pagination**
```sql
-- แบ่งหน้า 20 รายการ/หน้า
SELECT * FROM Advisor
WHERE project_capacity > 0
ORDER BY advisor_id
LIMIT 20 OFFSET 0;  -- หน้าที่ 1
```
